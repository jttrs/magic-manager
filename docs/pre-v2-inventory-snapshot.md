# Pre-V2 inventory snapshot

This is the V1.7 inventory state captured on **2026-05-30**, immediately before the V2.0 schema refactor (which splits the conflated `list_rows` table into separate `inventory`, `wishlist_entries`, `decks`, and `deck_cards` fact tables).

The data here serves as the **first re-ingest test case** for the V2 schema — after the migration runs, `mm query show 'inventory'` should show every card listed below with the same finishes and quantities. If anything diverges, the migration has a bug.

## Source

Generated by querying `list_rows` joined to `cards` for every row with `quantity > 0`, before the V4 migration was applied. The only label with non-zero qty was `owned:sld` (the V1.7 bulk-add of two Secret Lair drops).

Every other label in the V1.7 DB was `set:fin` seed data (the FIN family universe, ~2,181 rows all at qty=0). Those are re-derivable via `mm set sync fin` and intentionally not preserved across the migration — the V4 migration drops `set:*` rows where qty=0 because they're seed data, not ownership claims.

## Why this doc exists

After the V4 migration:

1. **Migration regression test** (Phase 6.1): assert that `inventory` has exactly 18 rows post-migration, all foil, with scryfall_ids matching the cards in this doc.
2. **Re-ingest test** (Phase 6.2): wipe the DB, re-run `mm bulk-add` for the SLD 1858–1872 + 7001–7003 ranges, and confirm `mm query show 'inventory'` shows the same 18 cards.
3. **Provenance**: months from now, when someone (probably future-me) is wondering "wait, did we actually preserve any V1 data through the V2 cutover?", this doc is the unambiguous answer: yes, these 18 cards.

If the V4 migration drops or alters any of these rows without intending to, that's a bug. This doc is the contract.
